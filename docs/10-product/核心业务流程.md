# 核心业务流程

完整产品边界见 [组织与账单产品设计](./组织与账单产品设计.md)。

## 个人账号与扣费流程

系统真实扣费主体始终是个人用户账号。

```text
请求 -> Token -> User -> 个人钱包/订阅 -> 预扣 -> 结算/退款 -> Log
```

组织扩展不进入这条链路，不新增组织预扣、组织结算、组织退款或组织消费流水。

## 创建组织

系统管理员在 `/admin/organizations` 创建组织：

1. 输入组织名称。
2. 后端创建 `Organization`，状态默认为启用。
3. 不创建任何 `OrganizationMember`。

创建组织不会让任何用户加入组织，也不会修改用户账号、余额、订阅、Token 或 `group`。

## 添加组织成员

系统管理员或组织 Admin 可以把已有用户加入组织：

1. 在组织成员页点击 `Add member`。
2. 搜索并选择一个已存在的启用用户。
3. 选择角色。
4. 后端校验该用户当前不属于其他组织。
5. 后端创建 `OrganizationMember`，记录 `joined_at`，写入 `billing_start_at = joined_at`，并保持 `left_at = 0`。

角色规则：

- 只接受 `admin`、`member`。
- `admin` 可以管理组织信息、成员和角色，并查看完整组织账单。
- `member` 只能查看自己的组织内用量。

## 移除组织成员

移除成员只结束组织成员关系：

1. Admin 触发二次确认。
2. 当前操作者可以移除自己，但如果自己是最后一个活动 Admin，则拒绝操作。
3. 后端在同一事务中校验组织仍有活动 Admin 并设置 `left_at`，不物理删除成员关系。

移除成员不删除用户账号，不修改 Token，不影响个人余额、订阅、充值、退款或历史日志。

## 组织账单查询

组织账单只解释已经发生的个人消费日志：

```text
Organization -> OrganizationMember.UserId -> Log.UserId
```

统计范围按成员有效期过滤：

- `effective_billing_start = billing_start_at > 0 ? billing_start_at : joined_at`
- `Log.CreatedAt >= effective_billing_start`
- `OrganizationMember.LeftAt = 0` 或 `Log.CreatedAt < OrganizationMember.LeftAt`

新成员默认从加入时间开始统计。经系统管理员或组织 Admin 确认后，可以通过“预览 → 应用”把当前成员的账单归属起点向前扩展，纳入仍保留的加入前消费；成员离开后，组织仍可查看其账单窗口内产生的历史消费。

## 历史账单归属回填

1. 管理员选择候选账单归属起点。
2. 后端预览相对当前窗口新增的请求数、quota、Token、日志范围和窗口冲突。
3. 管理员确认业务归属和新增金额。
4. 应用请求携带预览返回的当前起点作为乐观锁预期值。
5. 后端在事务内重做 CAS、同组织窗口不相交校验和增量计算，再更新 `billing_start_at` 并写审计。
6. 前端刷新 Summary、各维度、Logs、导出和 Invoice。

该流程不修改 `joined_at`、个人余额、订阅、Token 或原始消费日志。当前只允许向前扩展，不允许调晚起点截断已纳入消费。

## 组织 Invoice

```text
成员账单窗口
  -> 消费 Log.Quota
  -> 按成员、完整模型、北京时间月份聚合
  -> 模型分类
  -> 应用组织月度结算系数
  -> Invoice 页面与专用 CSV
```

Invoice 默认当前北京时间自然月，只统计消费日志。结算系数只改变组织结算报表的展示金额，不参与个人请求的预扣、结算或退款。
