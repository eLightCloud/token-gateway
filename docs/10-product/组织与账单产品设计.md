# 组织与账单产品设计

## 产品结论

系统采用“个人扣费、组织汇总、模型分类”的账单模型：

```text
个人账号 = 唯一真实扣费主体
组织      = 成员集合 + 报表权限 + 汇总视图
模型      = 消费分析维度，不是账号或付款主体
```

组织能力服务于企业用量治理，不改变现有个人钱包、订阅、API Key、预扣、结算和退款链路。

## 当前状态

基础组织管理与用量分析已经上线。本次候选版本新增独立 Invoice 计费账单和组织结算系数；代码与自动化验收已完成，MySQL、PostgreSQL、ClickHouse 实际环境验证和大数据量压测仍是发布前检查项。后续变更必须继续遵守“个人扣费、组织汇总”的边界。

## 个人账单能力

个人账号持有并承担：

- 钱包额度、已用额度和请求次数。
- 充值订单、兑换码充值和管理员补单。
- 订阅套餐、订阅实例、订阅额度及额度重置。
- `subscription_first`、`wallet_first`、`subscription_only`、`wallet_only` 四种扣费偏好，默认 `subscription_first`。
- API Key 额度限制和用量统计；API Key 仍归属于个人用户。
- 消费、退款、充值和管理日志。

模型倍率、固定价格和表达式阶梯计费决定个人请求的实际 `quota`；组织报表只读取已经形成的消费事实。

## 组织产品边界

组织负责：

- 组织名称和启用/停用状态。
- 成员关系及其有效期。
- Admin、Member 两类组织角色。
- 按成员、模型、渠道、日期和日志明细汇总用量。
- 在权限范围内查看和导出组织消费日志。
- 为系统管理员和组织 Admin 提供按模型类别、账号生成的 Invoice，并按组织自己的月度结算系数计算展示金额。

组织不负责：

- 余额、充值、订阅、付款客户、发票或退款。
- API Key、Token 归属或请求时的组织选择。
- 部门、团队、项目、成本中心或跨组织切换。
- 独立组织消费流水，或改写个人历史扣费事实。

`group` 继续表示用户分组、模型可用性、路由和倍率，不得在产品文案或界面中解释为组织。

## 生命周期

### 创建组织

系统管理员先创建空组织，只填写名称。创建后不会自动把创建人或任何用户加入组织。

### 添加成员

系统管理员在组织详情中搜索现有启用用户，并以 Admin 或 Member 身份添加到组织。

一个用户同一时间只能有一个活动组织成员关系。注册、登录、创建 Token、充值、订阅和 API 请求都不会自动加入组织。

新成员的账单归属起点默认等于真实加入时间。添加成员本身不自动吞并加入前的历史消费。

### 调整历史账单归属

系统管理员或组织 Admin 可以为当前活动成员显式向前调整账单归属起点：

- 先预览候选起点将新增纳入的消费请求数、quota、Token 和日志时间范围，再确认应用。
- 候选起点必须为正数、不得位于未来，也不得晚于当前生效起点；当前能力只补历史，不允许通过调晚起点截断已经纳入的消费。
- 同一用户在同一组织内的多段成员账单窗口不得重叠，避免同一日志重复统计；跨组织窗口可以重叠，各组织是独立报表视角。
- 历史日志已经被 TTL、清理任务或人工操作删除时，只汇总仍保留的事实，不生成估算数据。
- 调整只改变组织报表对日志的解释，不改写成员加入时间、个人余额、订阅、Token 或历史消费日志。

### 调整角色

- 系统管理员可维护所有成员关系。
- Admin 管理组织名称、成员和角色，并查看完整组织账单。
- Member 只查看自己在成员有效期内的用量，不查看其他成员。

### 移除成员

移除成员只结束成员关系并记录 `left_at`，不删除用户或历史日志，也不改变个人余额、订阅和 Token。

## 组织账单口径

组织账单按成员关系有效期解释个人日志：

```text
Organization -> OrganizationMember.UserId -> Log.UserId

effective_billing_start =
  billing_start_at > 0 ? billing_start_at : joined_at
```

- 未调整历史归属时，加入组织前的消费不进入组织账单。
- 管理员确认并向前调整账单归属起点后，起点至加入时间之间仍保留的消费进入组织账单。
- 成员有效期内的消费进入组织账单。
- 离开组织后仍保留其有效期内的历史汇总。
- 默认用量视图只统计消费日志。
- 对账口径可分别纳入退款和系统调整，不能把不同类型静默合并成净额。
- 消费金额由已落库的 `Log.Quota` 换算，是汇总、用户、模型、渠道、趋势和明细的首要账单指标；原始 quota 同时保留用于审计。
- 消费金额始终按真实货币展示：站点为 USD、CNY 或自定义货币时沿用其汇率，站点为 TOKENS 时使用 USD 等值，避免“金额”和原始额度重复。
- 模型同时展示当前计价规则；它只解释当前固定价格、倍率或阶梯计费配置，不重算历史消费。

### Invoice 与结算系数

Invoice 是组织用量分析之外的客户结算视图：

- 账期统一按北京时间，默认当前自然月，并支持最长 24 个自然月的自定义日期范围。
- 第一张表按模型类别 × 账号展示折前金额、类别合计、结算系数和结算后金额；第二张表按完整模型 × 账号展示折前金额和占比。
- 系数是组织级、按北京时间自然月生效的独立变量，默认 `1.0000`；允许 `0.0000` 至 `10.0000`，其中 `0.0000` 明确表示免费/结算豁免。
- 配置页只列出该组织成员账单窗口中实际产生过消费的模型类别；未知模型使用稳定的单模型 fallback 类别。
- 计算使用原始整数 quota，在类别 × 月份粒度应用四位定点系数，最终展示时再舍入；不得使用页面已格式化单元格反算。
- 修改历史月份会重新呈现组织 Invoice，但不会修改个人余额、订阅、Token、消费日志或原始扣费结果。

## 当前实现范围

- 管理员可以创建空组织、设置状态并维护成员。
- 成员角色只接受 Admin、Member。
- 活动成员关系在 SQLite、MySQL 和 PostgreSQL 上都保持单用户唯一。
- 新成员默认从加入时间开始统计；历史账号可通过“预览 → 应用”显式向前扩展账单归属窗口，成功与失败尝试都保留管理审计。
- Admin 的完整组织账单权限与 Member 的个人范围隔离。
- 汇总、趋势、成员、模型、渠道和明细使用同一组服务端筛选口径。
- 组织账单不触发任何预扣、个人扣费结算、退款，也不改写历史消费事实。
- 独立 Invoice 默认显示当前北京时间自然月，页面查询和专用 CSV 导出复用同一聚合与结算算法。
- 导出复用当前筛选条件；Billing 页面使用新增完整账单端点，汇总、用户、模型、渠道、趋势和明细都包含可计算的消费金额、币种和原始 quota，模型额外包含当前计价规则。
- Logs 页面使用独立的展示型日志导出端点，提供北京时间、用户、模型、消费金额、币种及输入/输出 Token；既有日志导出端点继续保持原始列结构和文件名。
- Logs 导出和 Billing 完整导出的明细均按有界批次流式输出，不再固定截断为 10,000 行。
- 展示型日志导出不包含请求标识和内容；兼容日志导出与完整账单导出仍属于管理员级排障导出。

## E2E 验收数据与用例

已上线版本采用 API E2E 验证核心业务合同。测试从 HTTP 请求进入真实 Gin 鉴权中间件和组织控制器，落到 GORM/SQLite 数据库，再校验 HTTP 返回与持久化结果；不模拟组织权限或账单聚合函数。

固定测试数据位于 [`controller/testdata/organization_e2e.json`](../../controller/testdata/organization_e2e.json)，包含：

- 1 个启用组织 `Acme AI Lab`。
- 1 个系统管理员、1 个组织 Admin、1 个组织 Member、1 个组织外用户和 1 个禁用用户。
- 两个仍归个人用户所有的 API Key，以及个人钱包额度、累计用量、请求次数和扣费偏好。
- Admin/Member 不同加入时间，加入前、加入后的消费日志，以及退款、系统调整和组织外用户日志。
- 两个模型、两个渠道和跨三个 UTC 日期的已结算消费事实。

未回填时，默认消费口径的固定期望值为：组织 `quota = 420`、请求数 `3`、输入 Token `360`、输出 Token `60`；Member 自己的受限视图为 `quota = 300`、请求数 `2`。对账视图额外纳入 `-30` 退款和 `+50` 系统调整，期望 `quota = 440`、记录数 `5`。组织外用户消费始终排除；加入前消费只有在管理员完成归属起点预览和应用后才进入组织账单，固定回填用例的组织结果为 `quota = 17,420`、请求数 `5`。

| 验收目标 | 自动化用例 | 关键断言 |
|---|---|---|
| 组织权限 | `TestOrganizationE2EPermissions` | 未登录拒绝；组织外用户无组织；Member 不能改组织或成员且只能看到自己；系统管理员可跨组织查看；Admin 可维护本组织；禁用用户不可加入；Root 可清理自己的历史成员关系；存在其他 Admin 时允许自退；最后一个 Admin 不可退出或降级 |
| 不影响当前功能 | `TestOrganizationE2ELeavesPersonalFeaturesUntouched` | 添加/移除组织成员和读取组织账单前后，个人钱包、累计用量、请求数、用户分组、扣费偏好、API Key 归属及 Key 额度均不变；个人 Token API 结果保持一致 |
| 账单正常 | `TestOrganizationE2EBillingScopesAndAggregatesSettledLogs` | 成员有效期、默认消费口径、对账口径、Member 强制自我范围，以及汇总、模型、渠道、趋势和明细结果一致；账单查询不修改个人余额或原始日志 |
| 历史归属回填 | `TestOrganizationE2EBillingStartBackfill`、`TestOrganizationE2EBillingStartMultiDimensionConsistency` | 预览后回填加入前日志，组织结果从 `420 / 3` 变为 `17,420 / 5`；Member 隔离、各维度勾稽、原始消费日志和个人余额保持不变 |
| Invoice 与结算系数 | `TestOrganizationE2EInvoiceAndSettlementFactor` | Member 被拒绝；Admin/组织 Admin 查询一致；默认系数、乐观锁冲突、结算金额和 CSV 导出一致 |
