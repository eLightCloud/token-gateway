# 验收标准

## 验收结论

基础组织管理、Billing 和 Logs 已完成验收并上线；历史归属起点和 Invoice 候选版本已完成自动化验收，仍待真实数据库、ClickHouse 和生产规模验证。以下条目是生产版本必须持续保持的回归合同。

## 组织管理

- 管理员能只输入组织名称创建空组织。
- 创建组织后不会自动添加任何用户。
- 管理员能将未加入组织的启用用户加入指定组织。
- 添加或修改成员角色时只接受 Admin、Member，其他角色必须被拒绝。
- 组织 Admin 能在用户侧成员页添加成员、修改角色和移除成员。
- Admin 能查看完整组织账单，Member 只能查看自己的组织内用量。
- Member 的账单查询只能覆盖自己在组织成员有效期内的用量，且前端必须提供对应的受限用量视图。
- 未加入组织的用户不显示组织入口，直接访问组织页面时显示未加入组织的空状态。

## 无侵入边界

- 管理员创建组织和创建系统用户时都不能隐式加入组织；如需加入组织，必须通过组织成员接口完成。
- 注册、登录、创建 Token、充值、订阅和 API 请求不会自动加入组织。
- 组织功能不修改用户 `group`、个人余额、个人订阅、Token 归属、扣费、退款或日志写入链路。
- UI 不出现组织余额、组织充值、组织订阅、组织 API Key、Header 组织归属或项目成本中心。

## 成员关系与账单

- 一个用户同一时间只能属于一个组织。
- 添加成员时必须拒绝已属于其他组织的用户。
- 移除成员只设置 `left_at`，保留历史成员关系。
- 组织账单按成员账单归属窗口汇总现有个人消费日志；旧成员 `billing_start_at = 0` 时回退 `joined_at`。
- 新成员的账单归属起点默认等于加入时间，因此未显式回填时，加入前历史消费不进入组织账单。
- 系统管理员和组织 Admin 可以先预览、再向前调整当前成员的账单归属起点；Member 不可操作。
- 回填应用必须校验候选时间、同组织窗口不重叠和 `expected_billing_start` 乐观锁；相同值重复应用保持幂等。
- 回填后 Summary、Members、Models、Channels、Trend、Logs、Billing 导出和 Invoice 使用同一成员窗口，且不修改个人余额、订阅、Token 或原始消费日志。
- 成员离开组织后，组织仍能查看其成员有效期内产生的历史消费。
- 组织明细导出复用当前筛选条件。
- 既有 `billing/logs/export` 保持单表日志 CSV、原始列顺序和文件名；完整账单通过新增 `billing/export` 提供。
- Logs 页面通过独立 `billing/logs/display-export` 导出可读时间、消费金额、币种、原始 quota 和合计 Tokens，不暴露请求标识与内容。
- 页面汇总、用户、模型、渠道、趋势和消费明细均以消费金额为首要指标，同时保留原始 quota。
- TOKENS 展示模式下消费金额使用 USD 等值，不得与原始 quota 重复。
- 导出的所有账单区块均包含纯数值消费金额、币种和原始 quota；模型区块额外包含当前计价规则。
- Invoice 默认按北京时间当前自然月查询，只统计消费日志，并按模型类别 × 月份应用组织结算系数。
- 结算系数必须以四位定点精度计算，允许 `0.0000` 至 `10.0000`；更新使用版本 CAS，冲突不得静默覆盖。
- Invoice 页面与专用 CSV 必须复用同一聚合结果，且专用 CSV 不包含日志内容、请求 ID 或上游请求 ID。
