# 上游同步与冲突解决指南

## 目标

本仓库采用以下同步原则：

> 功能、协议、数据、安全和工程能力以远端最新实现为基线；品牌、信息架构和视觉呈现以本地 `web/default` 设计为核心。

这不是“后端全部选 theirs、前端全部选 ours”。许多前端文件同时包含 API 调用、权限、状态管理、可访问性和视觉结构，必须按语义合并。

组织领域真源见 [组织与组织账单架构设计](../20-architecture/组织与组织账单架构设计.md)，界面真源见 [组织管理与组织账单 UI 设计](../90-ui-ux/组织管理与组织账单UI设计.md) 和 [默认前端品牌与视觉设计](../90-ui-ux/默认前端品牌与视觉设计.md)。分支差异属于临时快照，不作为长期同步依据。

## 真源矩阵

| 改动类型 | 默认真源 | 冲突处理 |
|---|---|---|
| API 协议、DTO、路由、状态码、provider 适配 | 远端 | 以远端为基线，调整本地调用方 |
| 数据模型、迁移、事务、计费、额度、安全 | 远端 | 以远端为基线；本地业务仅作为显式补丁重新适配 |
| 权限、鉴权、路由守卫、校验、错误处理 | 远端 | 保留远端行为和修复，不因 UI 改版删除 |
| React Query、缓存、请求状态、表单状态 | 远端 | 保留远端数据流，再接入本地组件结构 |
| 可访问性、键盘操作、reduced motion、响应式功能状态 | 远端功能 + 本地呈现 | 行为取远端，样式按本地设计调整 |
| 品牌名、Logo、favicon、首页默认文案 | 本地 | 保留本地，但不得破坏远端可配置品牌/自定义首页能力 |
| 首页区块、公共头尾、登录页视觉结构 | 本地 | 以本地 DOM/CSS contract 为核心，移植远端行为 |
| 色彩、字体、间距、圆角、阴影、动画 | 本地 | 保留本地 token 和组件样式，吸收远端新增状态 |
| locale JSON | 合并 | 远端 key 与本地 key 取并集；本地品牌文案优先 |
| `routeTree.gen.ts` 等生成文件 | 重新生成 | 不手工选边或拼接 |
| `AGENTS.md`、许可证、构建和依赖治理 | 远端/项目治理 | 不视为 UI 定制；完整保留远端约束并重新应用必要本地配置 |
| 临时设计稿、截图和过程文档 | 本地参考 | 不应阻塞功能同步；必要时归档 |

## 路径级默认策略

### 远端功能优先

以下目录或文件发生冲突时，先以远端版本为基线，再明确重放仍需要的本地业务：

```text
common/
constant/
controller/
dto/
i18n/                 # 后端 i18n
middleware/
model/
oauth/
pkg/
relay/
router/
service/
setting/
types/
go.mod
go.sum
```

数据库迁移必须保留双方新增内容并验证 SQLite、MySQL、PostgreSQL，不能整文件覆盖 `model/main.go`。

### 本地 UI 优先

以下内容优先保持本地设计意图，但仍要吸收远端功能与可访问性变化：

```text
web/default/public/lighting-*
web/default/src/features/home/components/
docs/90-ui-ux/
web/default/src/styles/index.css 中的 home-* / auth-* 样式域
```

### 必须语义合并

以下是高风险混合区，禁止按整个文件选边：

```text
web/default/src/main.tsx
web/default/src/components/layout/
web/default/src/features/auth/
web/default/src/features/organizations/
web/default/src/hooks/use-sidebar-*.ts
web/default/src/i18n/locales/*.json
web/default/src/styles/index.css
web/default/src/routes/
web/default/index.html
web/default/rsbuild.config.ts
AGENTS.md
```

主题或设置入口发生冲突时必须按能力拆分：固定浅色不自动等于删除字体、圆角、缩放、侧边栏、内容宽度或方向等布局设置。先列出设置抽屉承载的全部能力，再决定保留、迁移或删除。

## 推荐同步流程

### 1. 明确真正的远端与目标分支

```bash
git remote -v
git branch -vv
git fetch --prune <remote>
git log --oneline --decorate -n 20 <remote>/<branch>
```

不能仅凭远端名猜测：当前仓库可能只有 `origin`，也可能后续增加真正的 `upstream`。同步记录中必须写清远端 URL、目标 ref 和提交 SHA。

### 2. 清理工作区并建立恢复点

```bash
git status --short --branch
git diff --check
git switch -c backup/<branch>-before-sync-YYYYMMDD
```

未提交改动应按责任拆成独立提交，至少分为：

1. `feature:` 本地不可丢失的业务扩展。
2. `ui:` 品牌、布局和视觉定制。
3. `docs:` 说明和设计资产。

不要在脏工作区直接同步。若暂时不能提交，可使用带说明的 stash，但独立 WIP 提交通常更容易审查、回滚和重放。

### 3. 生成同步前清单

```bash
BASE=$(git merge-base HEAD <remote>/<branch>)
git log --reverse --oneline "$BASE"..HEAD
git diff --name-status "$BASE"..HEAD
git diff --stat "$BASE"..HEAD
```

将文件标为四类：`功能`、`UI`、`混合`、`生成/文档`。先分类，再开始合并。

### 4. 优先使用 merge 完成长分支同步

对于长期保留本地 UI 的分支，默认使用 merge，使“远端基线”和“本地定制”边界可追溯：

```bash
git switch <local-branch>
git merge --no-commit --no-ff <remote>/<branch>
git diff --name-only --diff-filter=U
```

只有在明确需要整理尚未共享的短提交历史时才 rebase。不要在 rebase 中凭直觉使用 ours/theirs：rebase 期间两者含义与普通 merge 的直觉相反。

### 5. 按语义解决冲突

#### 后端或功能文件

1. 先阅读远端改动原因和测试。
2. 用远端版本作为行为基线。
3. 将本地仍需要的组织/产品能力重新适配到新契约。
4. 保留双方新增迁移、路由和测试，不做整文件覆盖。

#### UI 混合文件

按以下顺序重建：

1. 远端 props、类型、API、hooks、权限、状态和事件处理。
2. 远端 loading/empty/error/disabled/focus/keyboard/mobile 状态。
3. 本地 JSX 信息架构、品牌、组件组合和 class contract。
4. 本地 CSS token、间距、颜色、阴影和动画。

最终结果应是“远端行为运行在本地界面中”，不是简单选一边。

#### 国际化

1. 先合并英文基准 key。
2. 执行 `bun run i18n:sync`。
3. 保留远端新功能翻译。
4. 对本地首页/品牌 key 使用本地确认文案。
5. 检查 7 个 locale，不提交未解释的英文回退。

#### 生成路由

1. 先解决 `src/routes/**`。
2. 删除 `routeTree.gen.ts` 的冲突标记，不手工拼路由节点。
3. 通过前端构建让 TanStack Router 插件重新生成。
4. 审查生成差异只包含预期路由。

### 6. 搜索静默丢失

冲突标记清零后继续检查语义丢失：

```bash
rg -n '^(<<<<<<<|=======|>>>>>>>)' .
git diff --check
git diff --name-status --cached
```

针对本地 UI 再检查：

```bash
rg -n 'Lighting|lighting-logo|home-|auth-' web/default/src web/default/public web/default/index.html
```

针对组织功能再检查：

```bash
rg -n 'Organization|organization' controller model router web/default/src
```

## 验证门禁

### 后端

至少执行：

```bash
go test ./model ./controller ./router
go test ./...
```

组织、计费、迁移或日志查询发生变化时，需要额外确认：

- 一个用户只能有一个活动组织成员关系。
- Admin/Member 权限边界未扩大。
- 成员有效期仍正确限定账单日志。
- 个人账号仍是扣费主体，组织不进入预扣/结算链路。
- CSV/日志导出遵循远端最新脱敏和权限规则。
- SQLite、MySQL、PostgreSQL 的迁移和查询都可用。

### 前端

从 `web/default/` 执行：

```bash
bun run i18n:sync
bun run typecheck
bun run lint
bun run format:check
bun run build:check
```

至少人工验证：

- 根路径默认首页及自定义 URL/HTML/Markdown 首页三种远端能力。
- 桌面和移动公共头部、登录跳转、通知、动态导航。
- 登录、注册、忘记密码、OTP、OAuth 回调和错误页。
- 管理员组织列表/详情，以及 Admin/Member 两类组织角色的页面可见性。
- 用量筛选、分页、导出、loading/empty/error 状态。
- 本地 Logo、标题、favicon、首页区块和浅色视觉。
- 键盘导航、focus-visible、reduced-motion 和常见分辨率。

### 文档

```bash
task docs:check
task ai:check
```

## 禁止做法

- 禁止在整个 `web/default` 上执行 `checkout --ours`。
- 禁止在整个后端上执行 `checkout --theirs` 后不审查本地必要业务。
- 禁止整文件覆盖 locale JSON、`model/main.go`、`router/api-router.go`、`index.css` 或 `AGENTS.md`。
- 禁止手工长期维护 `routeTree.gen.ts` 冲突结果。
- 禁止为了保留本地 UI 而丢弃远端安全、权限、校验、可访问性或错误处理修复。
- 禁止为了快速同步而把本地品牌和首页全部退回远端默认样式。
- 禁止把 merge 和 rebase 中 ours/theirs 的含义混为一谈。
