# 组织模块开发与验证指南

## 适用范围

修改以下任一内容时使用本指南：

- `model/organization.go`
- `controller/organization.go`
- `router/api-router.go` 中的组织路由
- `web/default/src/features/organizations/`
- 组织侧边栏、路由、国际化和账单导出

业务和架构真源分别见 [组织与账单产品设计](../10-product/组织与账单产品设计.md) 与 [组织与组织账单架构设计](../20-architecture/组织与组织账单架构设计.md)。本文件只说明以后如何安全修改和验证。

## 不可破坏的不变量

1. 个人用户始终是唯一扣费主体；组织查询不能进入预扣、结算、退款或 Token 鉴权。
2. 一个用户同一时间只能有一个活动组织成员关系。
3. 组织角色只允许 Admin、Member；旧 Owner/Billing 数据必须在迁移时归一。
4. 移除成员必须保留历史记录并清空 `current_key`。
5. 组织日志必须同时受成员有效期、角色范围和查询筛选限制。
6. Member 查询必须由后端覆盖为当前用户，不能信任客户端提交的 `user_id`。
7. 历史金额以 `Log.Quota` 为准，当前模型价格不能重算历史。
8. 主库和日志库不得假设可跨库 join。
9. 数据模型和查询必须兼容 SQLite、MySQL、PostgreSQL；日志聚合还需保留 ClickHouse 分支。
10. 导出必须复用与页面相同的权限和筛选条件。

## 后端修改顺序

### 成员关系

- 在 GORM 事务中读取组织、校验用户、检查活动关系并写成员。
- 活动关系写 `CurrentKey = strconv.Itoa(userId)`；离开时同时写 `left_at` 和 `current_key = nil`。
- 不使用物理删除，不通过 `users.group`、Token 或请求 Header 表达组织。

### 账单查询

- 先从主库读取成员关系段，再在 `LOG_DB` 应用 `applyOrganizationLogFilters`。
- Summary、Members、Models、Channels、Trend、Logs 的筛选语义必须保持一致。
- 退款或系统调整必须通过显式类型参数进入查询，不能改变默认消费口径。
- 聚合下推日志库；不要把全量日志取回 Go 后再过滤或按天分组。
- 明细分页必须先过滤组织范围，再全局排序和分页。

### API 与权限

- 用户侧继续使用 `/api/organization/current/*`，不要接受任意组织 ID。
- 管理员跨组织接口继续放在 `/api/admin/organizations/:id/*` 并使用 `AdminAuth`。
- UI 可见性只是体验优化，权限必须由控制器和模型重复校验。
- 修改组织状态语义前先决定它是否只是标签，还是要阻断成员查询、管理或个人 API 消费；不能只改一个入口。

### 导出

- 兼容日志 CSV 和完整六区块 CSV 是排障型导出，包含敏感字段。扩展客户导出时新建明确的列白名单，不要复用全量日志序列化。
- `billing/logs/export` 是兼容端点，必须保留原始单表列、列顺序和 `billing-logs.csv` 文件名；不能用完整账单替换它。
- Logs 页面调用独立的 `billing/logs/display-export`，按页面列导出可读时间、消费金额、币种、原始 quota 和合计 Tokens；浏览器通过 `timezone_offset` 传入显示时区偏移。
- 完整六区块账单使用新增 `billing/export`；Billing 页面调用完整账单端点。
- 保持 10,000 行上限或提供异步导出方案，避免无界查询。
- 任何脱敏、安全和权限修复必须同步覆盖用户侧与管理员侧导出。

## 前端修改规则

- 技术栈以 `web/default/AGENTS.md` 和 `components.json` 为准：React 19、TanStack Router、React Query、Base UI/shadcn 组合、Hugeicons、Tailwind CSS。
- API 与类型留在 `features/organizations/api.ts`、`types.ts`；页面组合位于该 feature，路由文件保持薄层。
- 查询 key 必须包含组织 ID、筛选和分页；成员变更后失效 self、members、billing 与管理员列表/详情相关查询。
- 所有可见文案使用 `useTranslation()` 和 `t()`，再从 `web/default/` 执行 `bun run i18n:sync`。
- 表格保留横向滚动、空状态、loading、禁用和错误反馈；图标按钮必须有可访问名称。
- 高风险操作使用确认对话框；角色修改失败必须恢复可理解状态。
- 产品目标与当前实现不一致时，在 UI 设计文档记录差异，不要把后端已经收紧的权限在前端扩大。

## 测试矩阵

### 后端行为

- 创建空组织，不自动创建成员。
- 添加启用用户并拒绝禁用/不存在用户。
- 并发或重复添加时仍只有一个活动组织。
- 添加和修改成员时只接受 Admin、Member，拒绝其他角色。
- 当前用户在存在其他活动 Admin 时可以移除自己；最后一个活动 Admin 不能退出或降级。
- 成员添加、移除和角色变更必须在组织行锁保护的同一事务中维护 Admin 不变量。
- 成员离开和重新加入时，账单只覆盖各自有效期。
- Summary、成员、模型、渠道、UTC 日趋势和跨成员日志分页结果一致。
- Member 查询被强制限制为自己。
- 独立日志库和 ClickHouse 排序/日期桶分支保持正确。

组织模型已有的行为回归集中在 `model/organization_test.go`；`controller/organization_export_test.go` 同时保护完整账单金额合同和旧日志 CSV 兼容合同。新增测试必须保护上述业务合同，使用 `testify/require` 和 `testify/assert`，不要锁定私有实现细节。

### 前端行为

- 未加入组织时无侧边栏入口，直达页面显示空状态。
- Admin/Member 两类角色分别验证菜单和页面。
- Member 只能看到自己的受限用量。
- 管理员可创建空组织、维护角色和查看三类详情 Tab。
- 筛选、刷新、分页和导出使用同一参数。
- 日期按 UTC 边界发送，用户、模型、渠道、趋势和明细都同时展示消费金额与 raw quota。
- 消费金额必须使用账单专用格式化：即使全局额度展示为 TOKENS，也显示 USD 等值，不能与 raw quota 重复。
- 模型价格列只标识当前计价规则；固定价格需要货币格式，不能暗示它是历史请求价格。
- CSV 的消费金额保持纯数值并单列币种，避免带符号或缩写后的字符串破坏 Excel 汇总。
- 桌面、窄屏、键盘焦点、空数据和请求失败状态可用。

## 验证命令

后端至少执行：

```bash
go test ./model ./controller ./router
go test ./...
```

前端从 `web/default/` 执行：

```bash
bun run i18n:sync
bun run typecheck
bun run lint
bun run format:check
bun run build:check
```

文档执行：

```bash
task docs:check
task ai:check
```

提交前再执行 `git diff --check`，并搜索组织旧术语、冲突标记和未翻译文案。
