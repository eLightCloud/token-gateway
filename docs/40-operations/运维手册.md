# 运维手册

## 组织账单变更发布

组织历史归属起点、Invoice、结算规则和流式导出会扩大日志查询范围，但不会修改个人扣费事实。发布前必须确认：

- 主库已备份，并记录当前应用版本、数据库版本和回滚镜像。
- MySQL、PostgreSQL 或 SQLite 的 GORM 迁移可以新增 `organization_members.billing_start_at` 和 `organization_billing_settlement_rules`。
- 独立 `LOG_DB` 或 ClickHouse 的连接、日志保留策略和最早可查询时间已确认。
- 所有网关实例使用同一代码版本和主库；滚动期间要求管理员暂停规则写入和历史归属回填，避免不同版本实例同时处理新写请求。
- 已选取内部组织作为灰度对象，并保存默认系数 `1.0000` 下 Billing 与 Invoice 的同边界对账结果。

推荐发布顺序：

1. 备份主库并记录日志库最早保留时间。
2. 先在测试环境或内部灰度实例发布候选版本并执行迁移；当前版本没有独立 Invoice 功能开关。
3. 验证组织成员、只读 Invoice、Billing、Logs 和个人请求链路没有退化。
4. 使用同一组 Unix 边界对比 Billing gross quota 与 Invoice gross quota；任何差异必须归因到成员窗口、日志类型、时区边界或日志保留。
5. 验证系统管理员、组织 Admin、Member 的权限矩阵。
6. 完成灰度后滚动发布全部生产实例；滚动完成前继续暂停结算规则配置和历史归属回填。
7. 在接近生产的数据量下验证页面 P95、同步 CSV 时长、网关超时和日志库负载，再通知管理员恢复写操作。

## 历史账单归属回填

### 操作前确认

只对经业务负责人确认属于目标组织的历史账号执行回填。逐成员记录：

| 字段 | 用途 |
|---|---|
| 组织、用户、成员记录 | 确认操作对象 |
| `joined_at` | 保留真实成员加入时间 |
| 当前生效 `billing_start_at` | 作为乐观锁预期值 |
| 候选归属起点 | 业务确认的历史边界 |
| 最早保留日志 | 判断日志库实际可恢复范围 |
| 新增请求数、quota、Token、日志范围 | 对账和审批 |
| 窗口冲突 | 阻断同组织重复统计 |

不得仅因为账号存在或日志仍保留，就把账号创建时间或最早日志时间批量写入所有成员。

### 预览与应用

当前组织 Admin 使用：

```text
POST /api/organization/current/members/:user_id/billing-start/preview
POST /api/organization/current/members/:user_id/billing-start
POST /api/organization/current/billing/billing-start/preview-batch
```

系统管理员使用对应的 `/api/admin/organizations/:id/...` 路径。

执行步骤：

1. 提交候选 `candidate_billing_start` 获取只读预览。
2. 核对新增请求数、quota、Token、最早/最晚日志、最早保留日志及 `conflict=false`。
3. 由业务负责人确认新增金额和归属边界。
4. 应用时同时提交候选值和预览返回的 `current_billing_start` 作为 `expected_billing_start`。
5. 成功后立即刷新 Summary、Members、Models、Channels、Trend、Logs、Billing CSV 和 Invoice。
6. 核对个人 `Quota`、`UsedQuota`、订阅、Token 和消费日志行数未变化，并保存管理审计记录。

同值重复应用是幂等操作。CAS 冲突时必须重新预览，不得用旧预览重试。已被 TTL 或清理任务删除的日志不能通过本流程恢复或估算。

### 错误回填处置

当前 API 只允许把归属起点向前扩展，不允许调晚起点；因此错误回填不能通过同一接口直接撤销。

- 立即停止该组织的后续规则调整和对外账单导出。
- 保存回填审计、预览结果和回填前后对账数据。
- 优先发布经过评审、具备审计和并发保护的修复能力；不要绕过服务层直接修改数据库。
- 若必须恢复备份，应进入维护窗口，先评估备份时间点之后其他主库写入的恢复影响，再执行完整恢复方案。
- 处置完成后重新校验全部账单维度、Invoice、个人计费事实和审计链路。

## Invoice 与结算规则验证

- 默认账期必须是北京时间当前自然月，自定义账期最长 24 个自然月。
- 未配置类别显示 `1.0000`；`0.0000` 明确显示免费/结算豁免。
- 历史、当前、未来月份规则分别验证，跨月账期应展示多档系数。
- 规则更新的 `expected_version` 冲突必须返回 409，不能覆盖新版本。
- Invoice GET 与专用 CSV 的账号列、类别合计、模型合计、折前 quota 和结算后金额必须一致。
- Invoice CSV 不得包含日志内容、IP、请求 ID、上游请求 ID 或 `other`。
- Billing/Logs 的明细导出必须按有界批次持续输出，不得固定截断；大账期仍需关注反向代理超时和客户端连接时长。

## 回滚

Invoice 和结算规则没有独立运行时功能开关。需要应用级回滚时：

1. 停止规则写入和历史归属回填操作。
2. 滚动部署到上一个已验证版本。
3. 保留新增列、结算规则表和历史规则版本，不删除审计事实。
4. 验证旧版本忽略新增表后，个人请求、Billing 和 Logs 仍正常。
5. 不清理或改写原始消费日志。

如果回滚原因是错误的成员归属起点，按“错误回填处置”执行；单纯回滚应用版本不会自动恢复已经写入的 `billing_start_at`。
