# 组织历史账单归属修复开发方案

## 文档目的

本文记录组织账单上线后发现的历史消费缺失问题、修复业务目标、数据口径、实施方案、迁移策略和验收标准，作为后续编码、生产回填和验证的当前开发基线。

## 问题背景

组织与组织账单功能上线前，平台已经存在账号及其历史消费日志。上线后，管理员创建组织并添加这些既有账号，组织账单只能看到成员被添加之后发生的请求，无法汇总这些账号此前已经落库的历史消费。

当前行为不是日志写入失败，也不是前端日期筛选失效，而是现有成员有效期口径的直接结果：

1. 添加成员时，`OrganizationMember.JoinedAt` 被写为当前时间。
2. 组织账单查询使用 `JoinedAt` 作为日志归属窗口的起点。
3. 实际查询条件为 `Log.CreatedAt >= max(JoinedAt, StartTimestamp)`。
4. 因此前端即使选择更早日期，也不能越过 `JoinedAt` 读取加入组织前的日志。
5. 现有单元测试、E2E 固定数据和产品文档都把“加入组织前的消费不进入组织账单”作为当前正确合同。

当历史账号在组织功能上线时才补建成员关系，`JoinedAt` 接近功能上线时间，因此表现为“组织账单只记录上线后的请求”。

## 修复的业务目标

### 核心业务目标

**让组织功能上线前已经存在、且经管理员确认属于该组织业务范围的账号，可以从指定的历史归属起点开始汇总其仍然保留在日志库中的历史消费，而不是只能从组织成员记录创建时间开始统计。**

修复后，组织管理员应能够看到经过确认的历史账号完整用量，包括汇总、成员、模型、渠道、趋势、明细和导出结果，并能明确知道这些数据从什么时间开始归属于组织。

### 必须同时满足的业务结果

1. 保留真实的成员加入时间，不能为了补历史账单而篡改成员关系审计事实。
2. 历史归属必须由管理员显式确认，不得把所有当前成员的全部历史消费自动划入组织。
3. 新增成员默认继续从实际加入时间开始统计，避免改变当前安全口径。
4. 同一用户在同一组织内的多段成员关系（如离开后重新加入）的账单窗口不能重叠，避免同一条日志在同组织内被两段成员关系重复统计。跨组织窗口允许重叠：一个用户可同时属于多个组织，各组织账单是相互独立的用量视角，同一日志出现在多个组织账单中属于预期行为，不做跨组织拒绝。
5. 修复只改变组织报表对已落库日志的解释，不改变个人钱包、订阅、Token、预扣、结算、退款和充值逻辑。
6. 历史金额继续以落库的 `Log.Quota` 为事实，不使用当前价格配置重算历史费用。
7. 所有组织账单视图和两类 CSV 导出必须使用同一归属窗口，不能出现页面、汇总和导出口径不一致。
8. 历史日志已经被 TTL、日志清理或人工操作删除时，系统必须明确只能汇总仍然保留的数据，不能生成无法审计的估算账单。
9. 修复必须保持 SQLite、MySQL、PostgreSQL 以及独立日志库、ClickHouse 日志分支兼容。
10. 修复应保持现有 NEWAPI 接口和导出合同稳定，新增能力采用可选字段或独立扩展接口。

### 非目标

本次修复不包含：

- 将组织改造成余额、订阅或付款主体。
- 把 `organization_id` 写入所有历史或未来日志。
- 修改现有个人扣费链路或账单偏好。
- 根据当前模型倍率、固定价格或阶梯表达式重算历史日志。
- 恢复已经从日志库删除且没有备份的数据。
- 自动判断某个账号过去的消费一定属于当前组织。
- 建设新的发票、应收、共享余额或组织级支付系统。

## 根因定位

### 数据模型根因

`OrganizationMember.JoinedAt` 当前同时承担两个不同概念：

- 成员关系真正建立的时间。
- 组织账单开始归属该用户日志的时间。

对于组织功能上线后新加入的普通成员，这两个时间通常一致；对于功能上线前就已经服务于该组织的历史账号，它们并不一致。现有模型无法表达这种差异。

### 查询根因

现有组织账单先从主库读取组织成员关系，再对每段关系在 `LOG_DB` 中按 `user_id` 和成员有效期查询（对应 `logMembershipBounds` / `applyOrganizationLogFilters`）：

```text
start = max(joined_at, start_timestamp)
end   = min(end_timestamp, left_at)            # left_at == 0 时不作为上限
created_at >= start
created_at <  end   当 end 取自 left_at        # 离开时间为开区间
created_at <= end   当 end 取自 end_timestamp  # 查询结束时间为闭区间
```

因此：

- 查询参数只能进一步缩小范围，不能把开始时间扩展到 `joined_at` 之前。
- Summary、Members、Models、Channels、Trend、Logs 和导出都会一致地排除加入前日志。
- 只修改前端默认日期无法解决问题。
- 原始历史日志仍然存在时，不需要复制或重写日志，只需要正确表达账单归属起点。

## 方案决策

### 拆分成员加入时间与账单归属时间

在 `OrganizationMember` 增加独立字段：

```go
BillingStartAt int64 `json:"billing_start_at" gorm:"bigint;default:0"`
```

字段语义：

- `JoinedAt`：成员关系实际创建时间，只用于成员关系和审计展示。
- `BillingStartAt`：该段成员关系的组织账单归属起点。
- `LeftAt`：该段关系的排他结束时间，保持现有语义。
- `BillingStartAt = 0`：仅用于兼容功能上线前的存量成员记录，查询时回退为 `JoinedAt`。新代码路径（新增成员、回填应用）永不写入 `0`：新增成员写 `JoinedAt`，回填写经校验的大于零的候选值。`0` 是迁移期过渡态，随存量记录被显式回填后自然消除。

`billing_start_at` 表达的是组织报表的用量归属，不表示组织承担实际扣费。

### 新的账单窗口

查询时先计算成员归属起点：

```text
effective_billing_start =
    billing_start_at > 0 ? billing_start_at : joined_at
```

最终日志窗口仅替换起点来源，结束边界语义与现状完全一致：

```text
start = max(effective_billing_start, start_timestamp)
end   = min(end_timestamp, left_at)
created_at >= start
created_at <  end   当 end 取自 left_at
created_at <= end   当 end 取自 end_timestamp
```

除开始时间来源外，现有日志类型、用户、模型、渠道、日期筛选、排序、分页和导出口径保持不变。

## 业务规则

### 新增成员

- 默认 `JoinedAt = now`，`BillingStartAt = JoinedAt`（新成员统一从加入时间开始计费）。
- 添加成员时不提供历史归属起点；如需补齐加入前的历史消费，添加后通过"调整账单归属起点"流程（预览→应用）单独设置，避免添加路径与回填校验（窗口不相交、CAS）耦合。

### 修改历史归属起点

允许 Admin 或系统管理员调整其有权管理的成员关系，但必须满足：

1. 时间戳大于零。
2. 时间戳不得晚于当前生效起点（`currentEffective`，≤ `JoinedAt`）；本次能力只允许向前补历史，禁止把已回填起点调晚以截断已纳入消费（否则预览会显示 Added=0 却实际移除历史）。
3. 时间戳不得位于未来。
4. 窗口不相交校验的范围限定为**同一组织内**该用户的全部成员记录（含离开后重新加入形成的其他记录）：同一条日志不得在同组织内被两段成员关系重复统计。**跨组织**窗口允许重叠——一个用户可同时属于多个组织，各组织账单是相互独立的用量视角，同一日志出现在多个组织账单中属于预期行为，不做跨组织拒绝。
5. 修改必须在事务内锁定相关成员关系并重新检查冲突。使用 `lockForUpdate` 锁定该组织成员记录：MySQL/PostgreSQL 发出 `FOR UPDATE`，SQLite 因不支持该语法而跳过（依赖单写者串行化），因此 SQLite 下仍必须在事务内重新执行冲突校验，不能假设行锁已阻止并发写入。
6. 操作审计必须记录组织、用户、成员记录、原起点、新起点、操作人、操作时间和结果。

### 历史归属起点的选择

管理员应根据真实业务关系选择以下策略：

- 从账号创建时间开始：仅适用于确认该账号自创建起就代表该组织使用平台的情况。
- 从指定业务日期开始：适用于账号早期有个人用途、迁移过程或组织实际成立时间晚于账号创建时间的情况。
- 从最早保留日志开始：仅作为管理员确认后的快捷选项，不能因为日志存在就自动认定其组织归属。

不能把 `User.CreatedAt` 或最早日志时间未经确认地批量写入所有成员。

### 移除和重新加入

- 移除成员仍然只写 `LeftAt`，不改写历史 `BillingStartAt`。
- 离开组织后，该段有效窗口内的历史账单继续保留。
- 重新加入会创建新的成员记录，默认从新的 `JoinedAt` 开始。
- 如果需要把组织外间隔也纳入报表，必须显式调整新记录并通过无重叠校验，不能自动吞并两段关系。

### 已知限制

本方案只向前扩展账单起点（`BillingStartAt`），不修改 `LeftAt`（离开时间是成员关系审计事实）。因此无法表达"成员有效期内中段存在个人用途需要从组织账单剔除"的场景——只能处理加入前的历史归属补齐，或加入前期的个人用途截断。若未来出现中段剔除需求，需要引入独立的报表归属结束字段，不属于本次范围。

## 后端改造方案

### 模型与迁移

1. 在 `OrganizationMember` 增加 `BillingStartAt`。
2. 保持 `OrganizationMember` 已有 AutoMigrate 入口，不引入数据库专属 DDL。
3. 不使用可能导致多数据库重复迁移的布尔默认值或数据库专属默认表达式。
4. 旧行保留零值，通过代码回退 `JoinedAt`，保证版本发布后账单总额不会自动变化。
5. 新增成员时由业务代码明确写入 `BillingStartAt = JoinedAt`。

### 模型服务

新增或调整以下稳定业务能力：

- 统一计算成员有效账单起点。
- 校验候选账单窗口是否与其他成员关系重叠。
- 在事务中更新账单归属起点。
- 预览某个候选起点将新增纳入的消费日志数量、quota、Token 数、最早时间和最晚时间。
- 批量预览组织内历史成员的待回填影响。

预览和正式汇总都必须复用同一日志类型口径。默认预览以 `LogTypeConsume` 为主，对账视图再分别展示退款和系统调整，不能把三类日志静默合并。

预览必须复用与正式汇总完全相同的窗口计算函数（`effectiveBillingStart` → `logMembershipBounds` → `applyOrganizationLogFilters`），仅把候选 `BillingStartAt` 代入起点，禁止为预览另写一套聚合 SQL，避免预览口径与正式查询在开/闭区间、`type` 默认值上漂移。

### API

保持现有组织、成员和账单端点不变，新增独立的历史归属扩展能力：

- 当前组织 Admin：预览和更新本组织成员的账单归属起点。
- 系统管理员：预览和更新任意组织成员的账单归属起点。
- Member：不能修改归属起点，只能按后端强制的本人范围查看结果。

接口至少需要返回：

- 当前 `joined_at`。
- 当前有效 `billing_start_at`。
- 候选 `billing_start_at`。
- 新增纳入的消费日志数量和 quota。
- 是否存在成员窗口冲突。
- 日志库当前可用的最早历史时间。

批量回填应采用“预览”和“应用”两步操作，应用请求必须带上预览对应的成员和候选时间，避免直接执行无法评估影响的全量修改。应用接口采用乐观锁：请求须携带预览时读取的 `expected_billing_start_at`（当前生效起点），服务端在事务内 `lockForUpdate` 锁定该组织成员记录后，先校验当前实际 `BillingStartAt` 仍等于该预期值（CAS），再重新执行同组织窗口不相交校验，最后写入新值。两个管理员并发应用同一成员时，后到达者因预期值不匹配而失败，不会静默覆盖。

### 审计

账单归属起点调整属于会改变报表历史结果的管理操作，应复用现有操作审计日志能力，至少记录：

- 操作 action（成功与失败分别用 `organization.billing_start_update` / `organization.billing_start_update_failed`）。
- 操作人和来源 IP。
- 组织 ID、用户 ID、成员记录 ID。
- 成功变更记录原值/新值与服务端重算的增量（`added_*`）；失败尝试记录候选值、预期值与失败原因（覆盖 CAS 失败、窗口冲突、向后截断被拒）。
- 结果区分：实际变更写成功审计，幂等重应用（相同值）不写，失败尝试写失败审计。

组织账单查询本身继续保持只读，不写入或修改原始消费日志。

现状核查表明“复用”并非零成本：`controller/organization.go` 当前对审计函数零调用，`auditContentTemplates` 与 `auditRouteActions` 均无 organization 条目，且 `/organization/current/members/*` 仅走 `UserAuth`、连中间件兜底审计都不触发。因此落地需要以下显式改造：

1. 在 `controller/audit.go` 的 `auditContentTemplates` 新增模板，例如 `organization.billing_start_update`: `Updated org ${organization_id} member ${target_user_id} billing_start from ${from} to ${to}`。
2. 在归属起点更新接口（当前组织 Admin 与系统管理员两条路径）显式调用 `recordManageAuditFor`：成功变更写 `organization.billing_start_update`（携带 `organization_id`、`target_user_id`、`member_id`、`from`、`to`、`added_*`），失败写 `organization.billing_start_update_failed`（携带候选值、预期值、错误原因）。`added_*` 由服务端在事务内重算，不接受客户端回传。
3. `/organization/current/*` 下的归属操作因不经 `AdminAuth`，必须由 handler 主动埋点并调用 `markAuditLogged`，不能依赖中间件兜底。
4. 回滚操作（恢复或置零 `BillingStartAt`）同样走上述审计路径，记录原值/新值。

## 前端改造方案

### 成员页面

成员列表同时展示：

- 加入时间。
- 账单归属起点。
- 离开时间。

对于 `billing_start_at = 0` 的兼容数据，界面显示实际生效的 `joined_at`，不能显示为 1970 年或空值。

### 添加成员

添加成员表单不提供历史归属起点（新成员统一从加入时间开始）。历史归属补齐由成员列表中的"调整账单归属起点"操作完成（见"历史回填"）。这避免添加路径静默忽略字段或与回填校验耦合。

### 历史回填

现有成员提供“调整账单归属起点”操作：

1. 选择候选时间。
2. 调用预览接口。
3. 展示新增日志数、消费 quota、时间范围和冲突结果。
4. 管理员二次确认。
5. 应用后刷新 Summary、Members、Models、Channels、Trend 和 Logs。

涉及冲突、日志缺失或历史范围过大的情况应展示明确错误，不允许前端自行忽略服务端校验。

所有新增用户可见文本必须进入现有 i18n 体系，并补齐 en、zh、fr、ja、ru、vi。

## 生产历史数据修复流程

### 第一步：确认日志仍然存在

在修改成员归属前，对目标账号检查：

- `LOG_DB` 是否仍有 `created_at < joined_at` 的消费日志。
- ClickHouse 是否设置 `LOG_SQL_CLICKHOUSE_TTL_DAYS`。
- 是否执行过日志清理任务。
- 最早可用日志时间和历史缺口。

如果日志已经删除，只能从备份或外部数据仓库恢复；本方案不会伪造缺失日志。

### 第二步：生成回填预览

按组织和成员生成以下清单：

| 字段           | 说明                               |
| -------------- | ---------------------------------- |
| 组织           | 当前目标组织                       |
| 用户           | 历史账号                           |
| 实际加入时间   | 当前`joined_at`                  |
| 候选归属起点   | 管理员选择的时间                   |
| 最早保留日志   | 当前日志库事实                     |
| 新增消费请求数 | 候选窗口比当前窗口新增的消费日志数 |
| 新增 quota     | 新增消费日志的 quota 合计          |
| 退款/调整      | 分类型展示，不静默合并             |
| 冲突           | 是否与其他成员窗口重叠             |

### 第三步：管理员确认

管理员逐组织确认：

- 账号过去是否真实代表该组织。
- 选择账号创建时间、业务日期还是最早保留日志。
- 新增纳入的账单金额是否符合历史台账。
- 是否存在个人用途或其他组织用途。

### 第四步：应用回填

只更新确认过的 `OrganizationMember.BillingStartAt`，不更新：

- `Log`。
- `User.Quota`、`UsedQuota`、`RequestCount`。
- `Token` 额度。
- 订阅、充值或兑换记录。
- 历史价格配置。

回填操作应可重复执行而不产生额外数据；相同目标值再次应用应保持幂等。

### 第五步：回填后对账

至少完成以下核对：

1. 组织汇总等于所有成员有效窗口内消费日志之和。
2. 成员维度之和等于 Summary。
3. 模型、渠道、趋势维度与同一批日志一致。
4. Logs 明细条数与汇总请求数一致。
5. 页面与完整 CSV、旧日志 CSV 使用相同筛选窗口。
6. 个人钱包、订阅、Token 和日志总行数在操作前后不变。
7. Member 仍然只能看到本人范围。

## 兼容性与性能

### 向后兼容

- 旧成员 `billing_start_at = 0` 时回退 `joined_at`，默认结果不变。
- 新字段和新接口均为增量扩展。
- 现有 `/billing/logs/export` 文件名、列和语义保持不变。
- 现有 `/billing/export` 六区块结构保持不变，只扩展纳入日志的有效时间窗口。
- 不改变错误响应和现有成员增删改接口的默认行为。

### 数据库兼容

- 主库变更通过 GORM 模型和现有迁移入口完成。
- 窗口冲突校验使用 GORM 查询和已有 `lockForUpdate` 约束。
- SQLite 不生成不支持的 `FOR UPDATE`。
- 不引入 MySQL、PostgreSQL 或 SQLite 独占语法。
- 日志查询继续保留 ClickHouse 的时间聚合、排序和分页分支。

### 性能

历史回填会扩大日志扫描时间范围，但不会改变当前“按成员关系查询、日志库聚合、Go 合并”的整体结构。

实施前后应在真实数据量下检查：

- 单成员长时间范围预览耗时。
- 大组织 Summary、Trend 和 Logs 耗时。
- 独立日志库和 ClickHouse 的过滤效率。
- 是否需要增加可移植的 `user_id + created_at + type` 复合索引。

索引调整必须基于实际执行计划和生产数据规模单独决定，不能为了本次逻辑修复直接在大日志表上盲目建索引。

## 验收标准

本修复只有在以下条件全部满足后才能视为完成：

1. 管理员能够为历史账号设置明确、可审计的账单归属起点。
2. 设置后，仍保留的历史消费能出现在所有组织账单视图和导出中。
3. 未设置历史归属的成员仍保持当前“从加入时间开始”的行为。
4. 同一条历史日志不能因组织迁移、离开重加或窗口重叠被重复统计。
5. 组织账单总额能够由原始日志按成员窗口独立复算。
6. 个人扣费、余额、订阅、Token、退款和日志写入完全不受影响。
7. 已删除的日志不会被估算或伪造，界面和回填结果能说明实际可用历史范围。
8. 现有账单接口、导出合同和 Member 数据隔离保持兼容。
9. SQLite 验证在 CI 通过；MySQL、PostgreSQL、ClickHouse 在对应环境变量就绪时通过（见“多数据库验证”）。
10. 产品、架构、UI、进度和变更文档完成同步。

## 回滚方案

本方案不修改原始日志和个人计费数据，因此回滚只需要恢复成员归属起点：

- 将目标成员的 `BillingStartAt` 恢复为原生效值（通常是 `JoinedAt`，须为大于零的正值——更新接口拒绝零值）。

回滚后组织账单将恢复为修复前的成员窗口口径，原始消费日志、个人余额和订阅数据不需要恢复。回滚同样走“审计”章节定义的归属起点更新审计路径，记录原值/新值与操作人。

## 当前结论

当前问题的正确修复方向不是取消成员有效期，也不是把所有成员的全部历史消费无条件划入组织，而是把“真实加入时间”和“组织账单归属起点”拆开。

该方案既能补齐经确认的历史账号用量，又能保留成员关系审计、跨组织隔离、现有个人计费链路和上游兼容边界，适合作为后续实施基线。
