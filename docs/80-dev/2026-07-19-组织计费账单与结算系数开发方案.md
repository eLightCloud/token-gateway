# 组织计费账单与结算系数开发方案

## 文档目的

本文定义组织“计费账单”能力的产品口径、数据模型、聚合算法、接口、页面、权限、迁移、测试和验收方案，作为后续编码与评审基线。

本文属于 `80-dev` 阶段性实施方案。当前实现合同已同步到：

- `docs/10-product/组织与账单产品设计.md`
- `docs/20-architecture/组织与组织账单架构设计.md`
- `docs/50-planning/组织与账单上线进度.md`
- `docs/90-ui-ux/组织管理与组织账单UI设计.md`

账单目标样式与示例数据见 `docs/10-product/账单表.md`。

## 实施结果（2026-07-19）

本方案已完成第一版实现：

- 新增独立的组织 Invoice 聚合、结算规则模型、北京时间账期解析、定点系数计算和 SQLite 自动迁移。
- 新增当前组织与系统管理员的 Invoice 查询、规则查询/更新、专用 CSV 导出、权限校验、版本冲突和管理审计。
- 新增管理员组织详情 Invoice Tab、当前组织 Admin 路由、动态账号交叉表、账期筛选、规则配置 Drawer 和七种前端语言文案。
- 新增模型单元测试和 Controller E2E，覆盖北京时间边界、默认/继承系数、零系数、负消费保护、首次创建并发、乐观锁、失败审计、Member 拒绝、Admin 查询和导出一致性。
- 组织日志导出与 Billing 多区块导出的消费明细均改为有界批次流式写出，不再截断为固定行数，也不把自定义账期内的全部日志同时保存在服务端内存。
- 已同步产品设计、架构设计、上线进度、UI 合同和产品示例；候选版本未冒充已经完成真实数据库与生产规模验收。

对 NEWAPI 原始代码只保留必要薄接缝：在主库迁移入口注册一张组织规则表、在组织路由组注册新端点、在既有审计模板表注册两个 action，以及在前端组织导航注册新入口。未修改 Relay、预扣/结算/退款、个人钱包、订阅、Token、`Log` 模型或消费日志写入链路。

SQLite 自动化测试、前端 typecheck、变更文件 lint 和 production build 已通过。MySQL、PostgreSQL、ClickHouse 实际环境一致性和百万日志压测仍属于发布前验证项，不能仅凭 SQLite 结果视为已经完成生产验收。

## 背景与现状

现有组织 Billing 页面已经提供汇总、成员、模型、渠道、趋势、明细和 CSV 导出，但它是用量分析页面，不是按“模型 × 账号”生成的结算账单。

现有后端分别提供：

- 按成员汇总。
- 按模型汇总。
- 按渠道汇总。
- 按日汇总。
- 分页消费日志。

现有接口不能直接、完整且高效地生成以下两张交叉表：

1. 模型类别 × 组织账号的归类结算汇总。
2. 完整模型 × 组织账号的消费汇总。

同时，现有模型价格、倍率和阶梯表达式属于个人请求实际扣费系统。组织账单不能修改或重新执行个人计费规则，只能读取已经落库的 `Log.Quota`。

### 已确认的现有代码事实

- 一个用户同一时间只能有一个活动组织成员关系：`OrganizationMember.CurrentKey` 使用用户 ID 形成唯一键，添加成员时还会在事务内检查现有活动关系。因此 `/organization/current` 可以唯一推导组织，不需要增加多组织选择器。
- 当前组织路由使用 `UserAuth`，组织 Admin 权限由 Handler 显式调用 `UserCanManageOrganization` / `UserCanViewOrganizationBilling` 收口；系统管理员组织路由使用 `AdminAuth`。新增规则写接口不能只依赖 `UserAuth`。
- `logMembershipBounds` 和 `applyOrganizationLogFilters` 是 `model` 包内的私有函数。新交叉聚合应继续位于 `model` 包或由 `model` 导出稳定领域能力，不能在 `service` 复制一套成员窗口算法。
- ClickHouse 日志库通过 GORM ClickHouse Driver 接入 `LOG_DB`。实现前组织趋势只有按 UTC 日分桶的方言分支；本次已把组织趋势统一改为北京时间自然日。

## 业务目标

### 核心目标

为组织提供独立的“计费账单”视图：

- 默认显示当前月。
- 支持自定义开始和结束日期。
- 按组织账号和完整模型展示折前消费。
- 按模型类别汇总并应用组织自定义结算系数。
- 结算系数属于组织，默认值为 `1.0000`。
- 配置页只展示该组织实际使用过的模型类别和模型。
- 系统管理员和组织 Admin 可以配置，Member 不可以配置。
- 账期统一使用北京时间（`Asia/Shanghai`，UTC+8）。

### 账单边界

组织结算系数只影响组织账单展示和账单导出：

- 不修改个人钱包、订阅、Token 或 API Key。
- 不参与请求预扣、结算和退款。
- 不修改原始消费日志。
- 不改变已经落库的 `Log.Quota`。
- 不使用当前模型价格重算历史消费。

组织账单仍然是“个人扣费事实的组织级汇总视图”，不是组织余额、发票、应收或付款系统。

### 非目标

本次不包含：

- 组织余额、充值、订阅、共享额度或统一付款主体。
- 税率、税票、发票编号、付款状态或应收账款。
- 修改全局模型倍率、固定价格或计费表达式。
- 为从未使用过的模型预先配置组织结算系数。
- 允许普通 Member 查看其他成员消费或修改组织规则。
- 恢复已经被日志 TTL 或人工清理删除的历史数据。

## 最小入侵原则

组织及组织账单属于本项目新增能力，可以在组织领域内部调整、拆分和补充测试；NEWAPI 原有个人计费、Relay、日志写入、钱包、订阅、Token、用户和全局认证能力必须保持最小入侵。

### 允许的改动范围

- 在 `model/` 新增独立的组织 Invoice、结算规则和聚合文件；可以复用或在组织领域内重构现有 `model/organization.go`。
- 在 `controller/` 新增独立的组织 Invoice Controller；可以复用现有组织权限和审计能力。
- 在组织路由组下追加新端点，不修改现有端点路径、参数或响应合同。
- 新增组织结算规则表并接入现有迁移入口。
- 在 `web/default/src/features/organizations/` 内新增 Invoice 页面、组件、Hooks、类型和 API。
- 为管理员组织详情增加 Invoice Tab，为当前组织 Admin 增加独立 Invoice 路由。
- 在组织功能自己的 React Query key 中加入认证用户 ID，并在组织功能范围内处理用户切换后的查询失效。

### 原则上禁止的改动

- 不修改 Relay 的预扣、结算、退款或供应商适配链路。
- 不修改钱包、订阅、Token、API Key、用户分组或个人计费主体。
- 不修改 `Log` 字段、消费日志写入流程或历史日志内容。
- 不把组织 ID、结算系数或类别写入 NEWAPI 原始请求和消费日志。
- 不修改全局模型倍率、固定价格、阶梯表达式或 `PriceData.OtherRatios`。
- 组织 Billing、Logs 和导出属于本项目新增领域，可以按北京时间展示和可用性要求演进；变更必须记录在组织文档和回归测试中。不得借此修改 NEWAPI 原始个人计费、日志写入或全局导出合同。
- 不为本功能重构全局认证 Store、全局 QueryClient 或通用路由框架；缓存隔离优先在组织 feature 内完成。
- 不向 `common/` 增加只有组织 Invoice 一个调用方的通用 helper。单用途计算留在组织领域文件中；只有形成稳定、可复用公共概念时才抽取。

### 例外处理

若实现确实需要修改 NEWAPI 原有公共代码，必须同时满足：

1. 组织领域内无法以组合、包装或新增文件完成。
2. 改动不改变原有调用方行为和公开合同。
3. 改动单独提交，并提供原有行为的回归测试。
4. 在代码评审中明确列出改动原因、影响范围和回滚方式。

验收时除验证新 Invoice 外，还必须运行现有个人计费、组织 Billing、日志、钱包、订阅和 Token 的相关回归测试，证明新增能力没有侵入原始扣费事实链路。

## 术语与计算口径

### 折前金额

折前金额是选定账期内符合组织成员归属窗口的消费日志额度之和：

```text
gross_quota = sum(Log.Quota)
gross_amount_usd = gross_quota / QuotaPerUnit
```

第一版计费账单固定使用 USD，避免站点展示货币或 TOKENS 模式改变账单合同。原始 `quota` 同时保留用于审计。

### 结算系数

结算系数是组织级、模型类别级变量：

```text
默认值       = 1.0000
小于 1       = 折扣
等于 1       = 原价
大于 1       = 加价
等于 0       = 免费/结算豁免
```

由于示例包含 `1.09`，页面统一使用“结算系数”和“结算后金额”，不使用容易误导的“折扣率”和“折后金额”。

`0.0000` 在本组织结算报表中允许使用，但必须展示为“免费/豁免”，保存时进行二次确认并写管理审计。它不修改个人实际扣费事实。

### 结算后金额

```text
settled_amount_usd = gross_amount_usd × settlement_factor
```

后端使用原始整数 quota 和定点系数计算，不能使用已经格式化为四位小数的表格单元格继续计算。

### 占比

完整模型表中的占比使用折前金额：

```text
model_share = model_gross_quota / organization_gross_quota
```

结算系数不改变完整模型消费表的占比。

## 表格合同

### 模型归类结算汇总

行表示模型类别，动态账号列表示该账号在该类别下的折前金额：

```text
类别 | 账号 A | 账号 B | ... | 折前合计 | 结算系数 | 结算后金额
```

规则：

- 类别单元格来自模型分类结果。
- 账号列只包含选定账期内有消费的组织账号。
- 空单元格按数值零处理，页面显示 `-`。
- 折前合计等于该类别全部账号列之和。
- 结算后金额由后端基于未舍入数据计算。
- 合计行的账号列和折前合计仍然是折前金额。
- 合计行最后一列等于所有类别结算后金额之和。

### AI 模型消费汇总

行表示完整模型名：

```text
模型 | 账号 A | 账号 B | ... | 合计 | 占比
```

规则：

- 完整模型名直接来自 `Log.ModelName`。
- 模型行按折前金额降序排列，再按模型名稳定排序。
- 账号列按账号折前总金额降序排列，再按 `User.Username` 稳定排序；`User.DisplayName` 只作为辅助展示。
- 最后一行展示每个账号的折前总额、组织折前总额和 `100.0%`。
- 空单元格显示 `-`。

### 精度与舍入

- 原始金额精度以整数 quota 为准。
- 结算系数使用四位定点精度。
- 聚合和乘法在后端完成。
- 页面金额统一显示四位小数。
- 导出金额至少保留六位小数，并保留原始 quota。
- 统一使用 half-away-from-zero 规则完成最终展示舍入。

示例中 `8.8873 × 1.09` 按已展示的四位小数计算会得到 `9.6872`，而示例写为 `9.6871`。实施时必须以原始日志 quota 复算；若原始数据不能得到示例结果，应修正示例，不能通过截断或特殊分支迎合展示值。

## 模型分类与可配置范围

### 第一版分类原则

第一版由平台提供稳定的、大小写不敏感的模型前缀分类，例如：

| 类别键     | 展示名   | 匹配前缀    |
| ---------- | -------- | ----------- |
| `claude`   | Claude   | `claude-`   |
| `gpt`      | GPT      | `gpt-`      |
| `gemini`   | Gemini   | `gemini-`   |
| `minimax`  | MiniMax  | `minimax-`  |
| `deepseek` | Deepseek | `deepseek-` |
| `kimi`     | Kimi     | `kimi-`     |

同一个完整模型只能命中一个类别。前缀规则按最长匹配优先，最后使用稳定类别键打破平局。

未命中已知类别的模型按完整模型名生成独立的 fallback 类别，默认结算系数为 `1.0000`，保证任何已产生消费的模型都可以出现在账单和配置页。

fallback 类别键按以下规则生成：

```text
normalized_model_name = lowercase(trim(model_name))
category_key = "model." + hex(sha256(utf8(normalized_model_name)))
```

要求：

- 哈希生成必须有确定性测试；类别键是稳定的、不透明的标识，不承担还原模型名的职责。
- 十六进制 SHA-256 只包含 URL、JSON 和索引安全字符，固定长度也避免超长模型名超过 MySQL 复合唯一索引限制。
- 原始 `Log.ModelName` 单独保留用于页面展示，不能用规范化名称覆盖原始名称。
- 规则更新把 `category_key` 放在 JSON 请求体而不是路径参数中。

平台内置类别展示名使用 i18n 键，例如 `organization.invoice.category.claude`；fallback 类别直接显示原始模型名，不进入 locale 文件。

### 组织可配置内容

组织只配置类别结算系数，不修改平台模型分类规则。配置界面展示：

- 类别名称。
- 当前组织实际使用过且命中该类别的完整模型列表。
- 当前生效系数。
- 生效月份。
- 恢复默认操作。

“只配置已使用模型”的服务端定义是：至少有一条属于该组织成员账单窗口的消费日志命中该类别。客户端不能提交任意未使用类别绕过此限制。

新模型首次产生消费后自动出现在配置页；没有显式配置时按 `1.0000` 结算。

## 结算系数数据模型

新增组织级结算规则表，建议模型名为 `OrganizationBillingSettlementRule`：

```go
type OrganizationBillingSettlementRule struct {
    Id                   int
    OrganizationId       int
    CategoryKey          string
    EffectiveMonth       int
    FactorScaled         int
    Version              int
    CreatedAt            int64
    UpdatedAt            int64
}
```

字段语义：

- `OrganizationId`：规则所属组织。
- `CategoryKey`：稳定模型类别键；fallback 类别使用前述 SHA-256 规则。数据库使用 `varchar(96)`，足以容纳当前最长 70 字节键，并降低 MySQL 复合唯一索引的长度风险。
- `EffectiveMonth`：北京时间自然月的数据库表示，格式为 `YYYYMM`，例如 `202607`。
- `FactorScaled`：结算系数放大 10,000 倍后的整数；`9900` 表示 `0.9900`。
- `Version`：从 `1` 开始的显式乐观锁版本，每次实际修改递增。
- `CreatedAt` / `UpdatedAt`：审计时间，不作为乐观锁依据。

约束：

- 唯一键：`organization_id + category_key + effective_month`。
- `factor_scaled` 由业务代码校验，允许范围为 `0` 到 `100000`，即 `0.0000` 到 `10.0000`。
- 请求体使用十进制字符串，例如 `"0.9900"`，服务端严格解析后转换为定点整数。
- 不接受 NaN、Infinity、负数、超范围值或超过四位的小数。
- `0.0000` 作为免费/结算豁免处理，不作为普通折扣静默保存。
- 默认值 `1.0000` 由业务代码回退，不为未配置类别创建冗余数据库记录。
- “恢复默认”写入从目标月份起生效的 `10000` 规则，不删除历史版本。
- API 的 `effective_month` 使用字符串 `YYYY-MM`；服务端必须校验四位年份、月份 `01` 至 `12`，再转换为数据库整数 `YYYYMM`。不接受 `2026-13`、`20260`、整数自动纠正或模糊日期。

迁移必须通过 GORM 模型完成，并同时兼容 SQLite、MySQL 和 PostgreSQL，不使用数据库专属 JSON、生成列或部分索引。

## 规则生效与历史账单

结算系数按北京时间自然月版本化：

- 新规则只能从某个月第一天开始生效。
- 查看某月账单时，使用 `effective_month <= 目标月份` 的最新一条规则；没有规则时回退 `1.0000`。
- 修改当前月规则会重新计算当前月账单。
- 修改未来月份不影响当前月和历史月份。
- 历史规则不物理删除。
- 当前没有“账单关账/锁定”模型，因此系统管理员和组织 Admin 可以修改历史、当前或未来月份规则；历史修改必须明确提示会重算历史账单并完整审计。若未来需要关账不可变性，应新增独立的账单快照和关闭状态，不得通过本方案暗示已经支持。

自定义范围跨越多个规则版本时：

- 折前表仍然按类别汇总。
- 结算后金额按月份分别应用当月系数后再求和。
- 如果所有月份系数一致，系数列展示该值。
- 如果存在多个系数，系数列展示“多档”，并在展开详情或 Tooltip 中展示月份与系数明细。
- API 返回 `factor_segments`，前端不自行推断历史规则。

## 账期与时间边界

### 默认当前月

计费账单首次打开时默认：

```text
timezone        = Asia/Shanghai（UTC+8）
start_date      = 当前北京时间月份第一天
end_date        = 当前北京时间月份最后一天
query_window    = [start_date 00:00:00+08:00, end_date 次日 00:00:00+08:00)
```

第一版计费账单固定使用北京时间，不依赖浏览器、服务器或数据库会话时区。Go 后端统一生成 UTC+8 的 Unix 秒边界，再把闭区间 `end_timestamp = next_day_start - 1` 传给现有组织账单过滤函数。

Invoice 是独立的北京时间账单合同。组织 Billing、Logs 页面和展示型导出也统一使用北京时间，但 Invoice 的 `start_date` / `end_date` 合同不复用日志导出的 `timezone_offset`。界面必须明确展示“账期（北京时间）”。

### 自定义时间

- 支持“本月”“上月”“自定义”。
- 自定义范围包含开始日和结束日。
- 开始日期不得晚于结束日期。
- API 使用严格的 `start_date=YYYY-MM-DD`、`end_date=YYYY-MM-DD`，由服务端按北京时间解释并返回最终 Unix 边界。
- 修改时间并点击“应用”后重新获取账单；可配置类别始终按该组织账单窗口内的历史实际用量确定，不被当前页面账期截断。
- 第一版时间状态保存在页面内，刷新后回到当前北京时间自然月；接口和导出始终使用同一组已应用日期。
- 导出复用页面相同的开始和结束时间。

## 聚合方案

### 数据源

继续使用现有组织账单真源：

```text
Organization
  -> OrganizationMember.UserId
  -> LOG_DB.Log.UserId
```

必须复用：

- `effectiveBillingStart`
- `logMembershipBounds`
- `applyOrganizationLogFilters`

从而保证计费账单、现有 Billing、Logs 和导出使用相同的成员归属窗口、日志类型和筛选边界。

第一版账单只统计 `LogTypeConsume`。退款和系统调整不静默混入模型消费；如后续建设对账账单，应作为独立区块展示。

### 新增长表聚合

后端新增“成员 × 完整模型 × 月份”聚合，返回内部长表：

```text
user_id
username
model_name
period_month
total_quota
request_count
```

禁止把全部原始日志下载到前端再聚合，也禁止依赖日志明细导出生成账单。组织日志 CSV 采用有界批次流式导出；Invoice 仍直接使用日志库聚合结果。

组织关系在主库、日志可能位于独立日志库或 ClickHouse，因此继续采用：

```text
主库读取成员及账单窗口
  -> 在 LOG_DB 内按成员窗口聚合
  -> Go 合并成员、模型和月份结果
  -> 应用模型分类与组织月度结算系数
  -> 返回完整账单 DTO
```

不得新增主库与日志库跨库 join。

月份分桶要求“北京时间语义一致”，不要求四种数据库使用相同 SQL 文本：

- Go 先生成每个北京时间月份对应的 Unix 秒半开区间，不能依赖数据库会话时区。
- 优先用这些 epoch 边界构造可移植的 `CASE` 分桶并按 `period_month, model_name` 聚合。
- 如果某个驱动对分组表达式支持不同，则按 `common.LogDatabaseType()` 生成方言分支；ClickHouse 仍通过现有 GORM Driver 执行。
- 禁止直接使用受 MySQL session timezone、PostgreSQL session timezone 或宿主机本地时区影响的日期函数。
- SQLite 与 ClickHouse 至少要用同一组边界数据验证结果一致。

当前实现按成员关系逐段查询，不存在大 `user_id IN (...)` 参数问题。若压测后改为批量成员聚合，必须对用户 ID 分批，建议每批不超过 500，并继续保持每段成员关系窗口不被合并丢失。

### 后端账单 DTO

建议新增：

```text
OrganizationInvoice
  period
  timezone = Asia/Shanghai
  start_timestamp
  end_timestamp
  currency = USD
  accounts[]
  category_rows[]
  model_rows[]
  gross_total_quota
  gross_total_amount_usd
  settled_total_amount_usd

CategoryRow
  category_key
  category_name
  models[]
  account_amounts[]
  gross_quota
  gross_amount_usd
  factor
  factor_segments[]

ModelRow

  category_key
  account_amounts[]

```

金额字段使用十进制字符串，quota 使用整数。所有合计和占比由后端生成，前端只负责展示。

## API 方案

### 当前组织

```text
GET /api/organization/current/invoice
GET /api/organization/current/invoice/settlement-rules
PUT /api/organization/current/invoice/settlement-rules
```

### 系统管理员

```text
GET /api/admin/organizations/:id/invoice
GET /api/admin/organizations/:id/invoice/settlement-rules
PUT /api/admin/organizations/:id/invoice/settlement-rules
```

查询参数：

```text
# Invoice 查询与导出
start_date=YYYY-MM-DD
end_date=YYYY-MM-DD

# 规则查询
effective_month=YYYY-MM
```

规则更新请求：

```json
{
  "category_key": "claude",
  "factor": "0.9900",
  "effective_month": "2026-07",
  "expected_version": 0
}
```

`expected_version` 用于乐观锁；新建规则时为 `0`。更新流程：

1. 事务内读取并锁定同组织、同类别、同生效月份规则。
2. 如果目标系数与当前系数相同，按幂等成功返回，不更新版本、不重复写审计。
3. 如果系数不同，要求 `expected_version` 与当前 `version` 一致。
4. 更新系数并把 `version` 加一。
5. 预期版本不匹配时返回明确冲突，不静默覆盖。

接口要求：

- 当前组织 ID 只能由登录用户活动成员关系推导。
- 现有活动组织关系具有单用户唯一约束；若数据损坏导致查到多个活动关系，接口必须报错，不能任意取第一条。
- 当前组织规则写接口必须显式校验组织 Admin。
- 系统管理员接口继续使用 `AdminAuth`。
- 服务端校验类别确实匹配该组织已使用模型。
- GET 账单与导出必须使用同一聚合服务，禁止前后端各自实现结算算法。

## 权限、安全与审计

### 权限

- 系统管理员可以查看和配置任意组织。
- 组织 Admin 可以查看和配置当前组织。
- Member 不能调用完整 Invoice 或规则写接口。
- 前端隐藏入口不能代替后端权限校验。

### 输入安全

- 系数必须由十进制字符串严格解析。
- 转换为 `factor_scaled` 前校验小数位数和范围。
- 禁止负数、NaN、Infinity 和超范围值。
- 金额乘法使用 `shopspring/decimal` 或等价定点整数算法，避免裸 `float64 -> int` 转换。
- 汇总不能产生负结算金额；发现异常 quota 时记录错误并拒绝生成不可审计账单。

### 审计

规则修改会改变组织账单结果，必须记录管理审计：

- action：成功为 `organization.settlement_rule_update`，失败为 `organization.settlement_rule_update_failed`。
- 操作人、来源 IP 和路由。
- 组织 ID。
- 类别键。
- 生效月份。
- 原系数和新系数。
- 请求 `expected_version`、读取到的 `actual_version` 和更新后的版本。
- 成功或失败及失败原因。

“幂等提交相同规则”严格指同一组织、同一类别、同一生效月份，目标系数与当前持久化系数相同；此时返回当前规则且不重复写审计。并发冲突和非法输入记录失败审计，但不写入规则。

## 缓存与性能

- React Query key 必须包含接口范围（当前组织或系统管理员）、组织 ID、当前认证用户 ID、开始日期和结束日期。角色不是缓存安全边界。
- 登录用户变化或退出登录时，由组织 feature 失效或移除自己的受保护查询；不得为本功能重构全局认证或 QueryClient。仅把角色加入 key 不能解决同角色账号切换后的旧数据残留。
- 规则更新后失效目标组织 Invoice、规则和相关导出缓存。
- 后端不缓存跨权限 DTO。
- 成员数、月份数、模型数和聚合查询数量必须记录压测结果。
- 页面只返回有消费的账号与模型，避免为大量零用量成员生成空列。
- 对超长自定义账期设置合理上限或异步导出之前，必须先完成压测；第一版建议限制为最多 24 个自然月。
- 不使用分页日志接口拼接账单。
- 组织日志导出以及 Billing 多区块导出的消费明细使用固定大小批次在后端归并并直接写入响应，内存占用与总日志条数解耦；流开始后的数据库或客户端断开错误只能终止响应并记录服务端错误，不能再改写为 JSON。

上线前压测基准至少覆盖：

```text
100 个成员关系窗口
24 个北京时间自然月
50 个完整模型
1,000,000 条匹配消费日志
```

初始性能目标为页面查询 P95 小于 2 秒、同步导出小于 30 秒。该目标必须在接近生产的日志存储和索引条件下验证；若不满足，先缩短同步账期上限或把导出改为异步任务，不能直接开放无边界长账期。

2. 运行一致性对账任务，验证默认系数 `1.0000` 与现有组织 gross quota 完全一致，并保存差异报告。
3. 再开放规则写接口和配置 UI。
4. 使用一个内部组织验证当月、上月、自定义账期和导出。
5. 最后向全部组织 Admin 开放入口。

### 回滚

- 隐藏 Invoice 入口并停用新增规则写路由即可回退功能。
- 新增规则表保留，不影响现有扣费和 Billing。
- 不删除已有规则版本，避免再次上线时丢失审计事实。
- 回滚不得修改或清理原始消费日志。

## 已知风险

- 示例账单只有四位展示精度，无法单独证明原始 quota 与结算后金额的最后一位。
- 自定义账期跨越多个系数版本后，单一“结算系数”列无法表达全部规则，需要“多档”展示。
- 大组织的查询量会随成员关系段数和月份数增长，必须在开放长账期前压测。
- 日志已被清理时只能基于仍保留的数据生成账单。
- 平台前缀分类需要随新模型家族维护；fallback 可以防止漏账，但展示名称可能不够友好。
- Invoice 固定使用北京时间；与其他页面对账时必须使用 Invoice 返回的准确 Unix 边界，不能只比较日期文本。
- 认证用户切换时若组织 feature 未失效自己的受保护缓存，可能短暂展示上一账号的旧数据；必须把组织范围缓存隔离作为前端验收项，但不得借此扩大为全局认证重构。
- 为追求复用而修改 NEWAPI 原有公共计费或日志链路会扩大回归面；实现评审必须检查新增逻辑是否真正隔离在组织领域。
